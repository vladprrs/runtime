  <action>
Create `Dockerfile` at repository root implementing a 3-stage build (per D-07, D-08). The Rust version in FROM tags MUST match rust-toolchain.toml (1.94.1).

Quality gate ordering follows Pitfall 4 from research: fmt (fast) -> clippy (medium) -> test (medium) -> build (slow). A failing gate stops the build immediately because each RUN is a separate layer.

```dockerfile
# Stage 1: Chef planner -- compute dependency recipe
FROM rust:1.94.1-bookworm AS planner
RUN cargo install cargo-chef --version ^0.1
WORKDIR /app
COPY . .
RUN cargo chef prepare --recipe-path recipe.json

# Stage 2: Build -- cache deps, run quality gates, compile
FROM rust:1.94.1-bookworm AS builder
RUN cargo install cargo-chef --version ^0.1
WORKDIR /app

# Cook dependencies first (cached layer -- only invalidated when deps change)
COPY --from=planner /app/recipe.json recipe.json
RUN cargo chef cook --release --recipe-path recipe.json

# Copy full source
COPY . .

# Quality gates (ordered fast-to-slow, each RUN fails the build independently)
RUN cargo fmt --check
RUN cargo clippy --workspace -- -D warnings
RUN cargo test --workspace
RUN cargo build --release --workspace

# Stage 3: Runtime -- minimal image with just the binary
FROM debian:bookworm-slim AS runtime
RUN apt-get update && apt-get install -y --no-install-recommends \
    ca-certificates \
    && rm -rf /var/lib/apt/lists/*
COPY --from=builder /app/target/release/runtime /usr/local/bin/runtime
EXPOSE 8080
ENTRYPOINT ["runtime"]
```

After creating the Dockerfile, run `docker build -t runtime-test .` to verify the full pipeline works. This will:
1. Install cargo-chef and compute the dependency recipe
2. Cache-cook all dependencies
3. Run `cargo fmt --check` (must pass)
4. Run `cargo clippy --workspace -- -D warnings` (must pass)
5. Run `cargo test --workspace` (must pass)
6. Run `cargo build --release --workspace` (must produce the runtime binary)
7. Copy the binary into a minimal Debian image

If the Docker build fails at any quality gate step, fix the underlying issue in the workspace code (not in the Dockerfile) and rebuild.

Note: The first Docker build will be slow (compiling all dependencies from scratch). Subsequent builds with only source changes will be fast due to cargo-chef caching the dependency layer.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Docker Hub registry | Base images (rust, debian) pulled from external registry |
| crates.io registry | cargo-chef and all workspace dependencies fetched during build |
| Dockerfile build context | Files sent to Docker daemon for build |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-06 | Tampering | Docker base images | mitigate | Pin exact version tags (`rust:1.94.1-bookworm`, `debian:bookworm-slim`) rather than `latest`. Version matches rust-toolchain.toml. Consider switching to digest-pinned images in production hardening phase. |
| T-01-07 | Tampering | cargo-chef installation | accept | Installed from crates.io with version constraint `^0.1`. Risk is low -- cargo-chef is widely used (1M+ downloads) and only runs during build, not in runtime image. |
| T-01-08 | Information Disclosure | Docker build context | mitigate | .dockerignore excludes `.git/`, `.planning/`, `.specify/`, `.claude/`, `specs/`, and `*.md` files. No secrets or planning artifacts enter the build context. |
| T-01-09 | Elevation of Privilege | Runtime container | mitigate | Minimal `debian:bookworm-slim` base with only `ca-certificates` installed. No shell abuse surface. Binary runs as non-root (Debian default for ENTRYPOINT). Future hardening: add explicit `USER nonroot` directive. |
| T-01-10 | Denial of Service | CI pipeline | accept | A malformed Cargo.toml or broken dependency could block all builds. This is by design -- quality gates MUST block deployment on failure. Recovery is fixing the code. |
</threat_model>
